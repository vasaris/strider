# «БРОДЯЖНИК» — ДВИЖОК СОЛО-НРИ С AI-ХРАНИТЕЛЕМ
## Архитектурный документ v1.0

> Рабочее название «Бродяжник» — в честь Strider Mode, на котором стоит MVP. Переименовать можно в любой момент, на архитектуру не влияет.

**Статус:** утверждённая конфигурация, готов к закладке репозитория.
**Контур:** приватный, некоммерческий. Ты + личные инвайты. Без публичных анонсов.
**Система MVP:** «Кольцо Всевластья» (TOR 2e) + «Игра для одного» + «Жизненные пути героев».
**Платформа:** PWA (Next.js), соло, текст без генерации изображений.

---

## §1. ВИДЕНИЕ И НЕ-ЦЕЛИ

**Что строим.** Персональный AI-Хранитель для соло-игры по «Кольцу Всевластья»: игрок ведёт героя через приключения в Эриадоре, движок честно считает правила и бросает кости, нарративный слой пишет прозу уровня «110/100» — конкретную, сенсорную, без слопа.

**Принцип №1: механика не живёт внутри LLM.** Все броски, проверки, таблицы, состояние — детерминированный код. LLM получает структурированные результаты и превращает их в повествование. LLM никогда не бросает кости, не выдумывает числа и не решает исходы проверок.

**Принцип №2: оракул — творческое ограничение, а не рудимент.** Таблицы ответов/преданий/удачи/неудачи остаются механикой движка. LLM интерпретирует их результаты, не заменяет их. Случайный результат «Разорвать + Братство» — это якорь, который не даёт модели скатиться в дефолтную колею.

**Критерии успеха MVP:**
1. Полное приключение (задание покровителя → путешествие → кульминация → фаза братства) проходится за 1–3 вечера.
2. Ноль механических ошибок: каждый бросок, каждое ЦЧ, каждый балл Изнурения сходится с книгой.
3. Прозаический вывод проходит Replacement Test и стоп-листы §0.2 анти-слоп протокола.
4. Состояние персистентно: можно закрыть вкладку и продолжить через неделю.

**Не-цели MVP:** мультиплеер; тактическая сетка; генерация изображений; кампания длиннее одного приключения (но схема состояния её поддерживает); любая монетизация; публичный доступ.

## §2. АРХИТЕКТУРА: ПЯТЬ СЛОЁВ

```
Игрок ⇄ UI (PWA)
        ⇅
   Orchestrator (цикл хода, §5)
   ├── 2.1 State machine  (Postgres: герой, мир, сцена, журнал)
   ├── 2.2 Rules engine   (чистая TS-библиотека, детерминированная)
   ├── 2.3 Narrative layer (Anthropic API + промпты + few-shot)
   ├── 2.4 Context compression (журнал фактов, суммаризация)
   └── 2.5 Eval harness   (LLM-судья, регрессии, golden transcripts)
```

### 2.1. State machine — схема состояния

Единственный источник правды. JSON в Postgres, версионируемый патчами (каждый ход = патч, история откатываема).

**hero:**
- Характеристики: СИЛА, СЕРДЦЕ, РАЗУМ + ЦЧ каждой (формула соло: **18 − рейтинг**)
- Навыки (рейтинги 0–6, флаги «любимый»), Боевые умения
- Доблесть, Мудрость; Награды и Особенности (с пометкой «бесполезна соло» из ИдО)
- Отличительные качества (включая обязательного **Странника**)
- Выносливость / Изнурение / текущая, Надежда / текущая, **рейтинг братства = 3 + бонусы**
- Тень: баллы, шрамы Тени, состояние (Угнетение и т.д.); состояния: ранен, изнурён
- Снаряжение (оружие со свойствами, броня, Знаменитые предметы → влияют на Бдительность Ока)
- Опыт: баллы навыков, баллы приключений, журнал достигнутых вех
- Культура, призвание, предыстория (из «Жизненных путей»), образ жизни

**world:**
- Календарь: сезон, счётчик Фаз братства (Йоль каждую ~третью)
- Текущий регион + тип местности (Пограничные / Дикие / Тёмные земли)
- **Бдительность Ока:** начальный рейтинг (по культуре, Доблести, Знаменитым предметам), текущий, порог преследования (18/16/14 + модификаторы)
- Покровитель, активное задание (текст + открытые вопросы из таблицы), статус
- Безопасное место (описание, обитатели)
- Флаги мира и последствия сцен обнаружения «о которых герой пока не знает»

**scene:**
- Тип: путешествие / бой / Совет / свободная / фаза братства
- Бой: участники, стойки (передняя / открытая / защитная / дальняя / **манёвренная**), Решимость/Ненависть врагов, раунд
- Совет: рейтинг сопротивления (3/6/9), накопленные успехи, провалы (лимит 3)
- Путешествие: пункт назначения, длительность (дни), пройдено, лог сцен пути

**log:**
- Журнал фактов (экстракция после каждой сцены: NPC, места, обещания, находки)
- NPC-память (кто встречен, отношение, что знает о герое)
- Сжатые саммари сцен (для context compression)

### 2.2. Rules engine

Чистая TypeScript-библиотека без побочных эффектов: `(state, action, rng) → (result, statePatch)`. **RNG сидируемый** — каждый бросок воспроизводим, что критично для тестов и evals.

Реализуемые подсистемы (источник: книга правил + ИдО):

| Подсистема | Содержание |
|---|---|
| **Кости** | Кость испытания (d12: 11 = Око, 12 = руна Гэндальфа), Кости успеха (d6, 6 = знак успеха), благополучный/злополучный бросок, вдохновлённые проверки |
| **Проверки** | ЦЧ = 18 − характеристика; степени риска (обычная / опасная / безрассудная → исход провала); особые успехи (траты знаков: узнать, бесшумно, быстро, влияние, преимущество) |
| **Оракулы** | Таблица ответов (5 вероятностей; руна = «да+», Око = «нет+»); таблица преданий (Действие/Аспект/Фокус, 12×6); таблицы удачи/неудачи (триггер: руна/Око на проверках) |
| **Око Мордора** | Стартовый рейтинг; рост (+1 за Око вне боя, +N за баллы Тени, +2 за Око в таблице неудачи); порог по местности; сцена обнаружения при достижении; сброс после сцены и в Фазе братства |
| **Путешествия** | Таблица сцен пути (бросок по местности), 7 типов сцен с таблицами деталей, последствия и Изнурение, значимые встречи как эскалация |
| **Бой** | Стойки, манёвренная позиция (только дальний бой, −1к врагам ближнего боя, задача «Продвинуться», механика выхода из боя), особый урон, логика врагов (выбор оружия по стойке, Способности через таблицу ответов) |
| **Советы** | Рейтинг сопротивления через оракул при неуверенности; вступление + взаимодействие; N успехов до 3 провалов |
| **Прогрессия** | Вехи (10 типов → баллы навыков/приключений), Йоль (+РАЗУМ баллов), траты в Фазе братства |
| **Фаза братства** | Восстановление, исцеление шрамов, снятие Тени по значимости подвигов (1/2/3), начинания, встреча с покровителем |

### 2.3. Narrative layer

**Вход:** структурированный пакет от оркестратора — намерение игрока, результаты бросков, результаты оракулов, патч состояния, релевантные чанки лора (RAG), последние факты журнала.

**Выход:** проза + (опционально) уточняющие вопросы игроку в духе Хранителя.

**Системный промпт** = TTRPG Architect §0.2 (анти-слоп, оба стоп-листа), адаптированный под:
- Толкиновский регистр: сдержанная героика, конкретика природы и быта, БЕЗ пастиша («древнее зло пробуждается» — в стоп-лист ВК-пака отдельной строкой; жанровые штампы Средиземья — главный слоп-риск)
- Перевод Пандора Бокс как терминологический канон (Хранитель, Изнурение, баллы Тени, Кость испытания...)
- Голоса NPC по культурам: хоббит ≠ гном ≠ дунэдайн (правила §0.2.3)

**Жёсткие контрактные правила промпта:**
1. Числа, кости, исходы — только из пакета движка. Запрещено упоминать механику, которой нет в пакете.
2. Результат оракула обязателен к вплетению. Если выпало «Жуткая сцена или зловещий знак» — в прозе он есть, конкретный и здешний.
3. Чувства героя не описываются — только то, что он воспринимает (§0.9).
4. Длина калибруется по типу: сцена пути 400–800 знаков, значимая встреча 800–1500, бой по раундам 300–600.

**Few-shot банк:** 15–30 эталонных пар «пакет движка → проза» из твоих материалов и вручную написанных образцов. Версионируется вместе с промптом.

### 2.4. Context compression

- После каждой сцены: экстракция фактов в журнал (структурированно: NPC / место / обещание / находка / угроза) + саммари сцены ≤ 400 знаков.
- В контекст нарративного слоя идут: системный промпт, few-shot, текущее состояние (компактный рендер), журнал фактов, саммари последних N сцен, полный текст только текущей сцены.
- Бюджет на ход: цель ≤ 15–20k токенов входа. Полная история никогда не передаётся.

### 2.5. Eval harness

- **LLM-судья** по рубрике §0.7 (специфичность, точность, играбельность, агентивность, тон, анти-слоп) — порог 80+.
- **Детерминированные проверки:** стоп-листы §0.2.1 (оба языка + ВК-дополнение) как regex/словарные тесты; терминологический контроль (запрещённые кальки: «хиты», «спасбросок» — здесь иная система!).
- **Golden transcripts:** 5–10 эталонных прохождений сцен; каждое изменение промпта — прогон и diff-оценка.
- **Механические юнит-тесты** rules engine на сидированном RNG: таблицы дают книжные распределения, Око растёт по правилам, ЦЧ считается верно.

## §3. КОНТЕНТ-ПАЙПЛАЙН

**Источники:** книга правил (242 стр., извлечение проверено), «Игра для одного» (32 стр.), «Жизненные пути» (12 стр.).

**Этапы:**
1. **Извлечение.** pikepdf (снятие RC4) → pdfplumber с колоночной сегментацией → постобработка (склейка переносов, чистка декоративных заголовков).
2. **Структурирование.** Таблицы → JSON (схема на таблицу: probability ranges, эффекты, ссылки); правила → атомарные карточки правил с цитатой-источником (стр.); противники → статблоки JSON; лор (Эриадор, регионы, покровители) → RAG-чанки 300–600 токенов с метаданными региона.
3. **Валидация.** Каждый JSON-файл проходит твою ручную сверку с книгой до пометки `verified: true`. Движок отказывается грузить неверифицированные таблицы в проде.

**Структура пака:**
```
content-packs/kv/
├── manifest.json        # система, версия, зависимости, verified-статус
├── mechanics/           # карточки правил (проверки, бой, путешествия...)
├── tables/              # все таблицы: оракулы, сцены, события, задания
├── adversaries/         # статблоки противников
├── lore/                # RAG-чанки: регионы, NPC, покровители
├── lifepaths/           # Жизненные пути: культуры × предыстории × события
└── tone.md              # тоновый контракт: регистр, стоп-лист пака, голоса культур
```

**Граница контуров:** `engine/` не содержит ни одной строки контента ВК. Любая Daggerheart/будущая система — новый пак с тем же интерфейсом манифеста.

## §4. ГЕНЕРАТОР ПРИКЛЮЧЕНИЙ

Автоматизированный Route C: скелет до игры, сессия по скелету.

**Сид приключения:**
1. Покровитель (выбор игрока или бросок) × задание (таблица 6×6 из ИдО) — задание уже содержит 2 открытых вопроса.
2. Ответы на открытые вопросы: броски по таблице преданий + LLM-интерпретация в контексте лора региона.
3. Регион и пункт назначения (карта Эриадора из книги правил, гексы и местности).

**Скелет (JSON, генерируется и проверяется до старта):**
- Завязка (встреча с покровителем — диалоговая сцена)
- Маршрут: дни пути, местности, ожидаемое Изнурение
- 2–3 ключевые сцены-якоря (значимая встреча, Совет, столкновение) с конкретикой: кто, где, чего хочет, секрет
- Антагонист/угроза с мотивацией и слабостью (правила NPC §4.2 Architect)
- Три исхода: победа / пиррова победа / провал — каждый с последствиями для мира и Тени
- Давление: Око Мордора работает автоматически; для сюжетного давления — дедлайн задания, если уместен

**Гарантии генератора (валидатор скелета):** три исхода существуют; у каждого NPC есть мотивация и секрет; якорные сцены не образуют рельсы (между ними — свободное пространство решений); открытые вопросы задания получили конкретные ответы.

**Во время игры** скелет — ориентир, не сценарий: оркестратор сверяется с якорями, но игрок волен идти мимо; оракулы и сцены обнаружения вбрасывают незапланированное.

## §5. ИГРОВОЙ ЦИКЛ ХОДА

```
Ввод игрока (свободный текст или кнопка UI)
  → Классификатор намерения:
     • вопрос «да/нет» о мире        → таблица ответов (вероятность: LLM предлагает, движок бросает)
     • открытый вопрос о мире        → таблица преданий → LLM-интерпретация
     • рискованное действие          → определение навыка + степени риска → проверка
     • движение к цели через глушь   → процедура путешествия (сцены по дням)
     • разговор с весомой просьбой   → Совет (рейтинг сопротивления)
     • бой (атака/нападение врагов)  → боевой цикл по стойкам
     • бытовое/безопасное действие   → без бросков, сразу нарратив
  → Rules engine: броски, таблицы, патч состояния
  → Триггеры: руна/Око → таблицы удачи/неудачи; рост Ока → проверка порога → сцена обнаружения; веха достигнута → начисление
  → Narrative layer: пакет → проза
  → UI: текст + анимация костей + обновление трекеров + патч журнала
```

**Правило прозрачности:** игрок всегда видит, что выпало и почему (бросок разворачивается в UI: кости, ЦЧ, модификаторы). Доверие к честности движка — продуктовая фича №1, отличающая нас от AI Dungeon.

## §6. UI (PWA)

- **Лента повествования** — основной поток: проза, врезки бросков, реплики NPC. Типографика под чтение (это продукт про текст).
- **Панель костей:** Кость испытания d12 с рунами + пул d6 Костей успеха. Анимированный бросок — зрелище; руна Гэндальфа и Око визуально события.
- **Лист героя** (выезжающая панель): характеристики с ЦЧ, навыки, снаряжение, качества.
- **Трекеры всегда на виду:** Выносливость/Изнурение, Надежда, Тень (+шрамы), **Бдительность Ока с порогом** — глобальные часы напряжения как элемент интерфейса.
- **Доска боя:** ряды стоек (передняя / открытая / защитная / дальняя / манёвренная), токены героя и врагов, перемещение = анимация между рядами. Никакой сетки — чистый TOR.
- **Журнал:** факты, задание с открытыми вопросами, лог путешествия по дням.
- Тёмная тема; визуальный язык твоих React-трекеров. Frontend-design skill — обязательное чтение перед вёрсткой.

## §7. ОНБОРДИНГ

1. **Выбор пака** (в MVP один — ВК; интерфейс выбора закладывается сразу).
2. **Герой через «Жизненные пути»:** три анимированных броска — культура (d6) → предыстория (d6: характеристики, любимый навык, качества) → основное событие (d12 с рунами: механический твист). LLM разворачивает выпавшее в персональную виньетку (имя, конкретные детали — с участием игрока). Альтернатива: ручной выбор из тех же таблиц. Соло-правки применяются автоматически: 15 баллов опыта, Странник, братство 3, ЦЧ от 18, без Важного товарища.
3. **Призвание** (подсказка: вестник / искатель сокровищ / страж рекомендованы для соло — но любое).
4. **Покровитель + первое задание** (таблицы ИдО; бросок или выбор).
5. **Безопасное место:** вопрос игроку («почему это не только убежище, но и дом?») — ответ в журнал, нарративный слой обязан к нему возвращаться.
6. **Тоновый контракт** зашит в пак (толкиновский регистр, 14+), без анкеты. Настройки границ — в опциях, не в онбординге.

## §8. СТЕК И РЕПОЗИТОРИЙ

- **Frontend:** Next.js (App Router) PWA, TypeScript, Tailwind; анимация костей — CSS/Framer Motion (3D-физика — поздняя полировка).
- **Backend:** Next.js API routes / server actions; движок — отдельная библиотека.
- **БД:** Postgres (Supabase): состояние, патчи, журнал, транскрипты.
- **LLM:** Anthropic API; модель нарратива — Sonnet-класс по умолчанию, A/B через eval.
- **Хостинг:** Vercel или Railway (опыт есть).

```
brodyazhnik/
├── engine/          # rules engine: чистый TS, ноль контента, сидируемый RNG
├── content-packs/
│   └── kv/          # см. §3
├── app/             # Next.js PWA
├── orchestrator/    # цикл хода, классификатор, сборка пакетов для LLM
├── prompts/         # системные промпты, few-shot банк (версионируются)
├── evals/           # судья, стоп-листы, golden transcripts, юнит-тесты
```

Без дедлайнов; этапы закрываются по критериям выхода, не по календарю.

**Гейт отложенного.** `docs/DEFERRED.md` — единственный канонический реестр отложенной работы. Критерий выхода этапа N не считается выполненным, пока в реестре открыт пункт с `DUE ≤ N`. Стартер каждой сессии этапа обязан эхо-вывести открытые пункты своего этапа из `DEFERRED.md`, а также указать, какой чат/подчат `docs/ROADMAP_SESSIONS.md` он закрывает; роадмап обновляется на каждом хендоффе (отметка чата выполненным, переразбивка ближайшего незакрытого этапа при расхождении плана с реальностью). Памяти и разрозненных `HANDOFF_*.md` для этого недостаточно — они дрейфуют между сессиями; реестр в репо не дрейфует. Новый отложенный пункт заносится в реестр в той же сессии, что и решение отложить.
